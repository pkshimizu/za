# za:hunk-resolve — hunk の全コメントを一括対応してコミット

[hunk](https://www.hunk.dev/) は「レビュー起点の diff ビューア」で、diff の特定の hunk / 行に
インラインでレビューコメントを付けられる。このスキルは、実行時点の**全コメントを取得**し、
各コメントの意図を汲んで**コード修正まで対応**し、終わったら**対応内容を一覧表示**、
**対応済みコメントを削除**、**git コミット & push** までを一気に行う。

狙いは、レビューで付いたコメントをまとめて反映し、追跡可能な形（コミット）に落とし切ること。

## 前提と制約（先に理解する）

- **コメントは起動中の hunk セッション内にのみ存在する。** `hunk diff` / `hunk show` などで
  開いた TUI（またはそのデーモン）が生きている間だけ `hunk session comment list` で取れる。
  セッションを閉じるとコメントは破棄される。
- これは**一括処理**（常駐監視ではない）。実行した時点で付いているコメントだけを対象にする。
- ユーザーへ質問・確認するときは、原則 `AskUserQuestion` で**選択式**で尋ねる。

## 手順

### 1. 全コメントを取得する

対象は原則カレントリポジトリ（`git rev-parse --show-toplevel`）。ユーザーが別パスを
指定した場合はそれを使う。次で**全コメント**を JSON で取得する。

```
hunk session comment list --repo <repo> --type all --json
```

各コメントの `noteId` / `filePath` / `newRange`（対象行）/ `body` / `hunkIndex` / `source` を把握する。

- アクティブセッションが無い（`hunk: No active session...`）場合: 「対象リポジトリで hunk を
  起動してコメントできる状態にしてください」と伝えて終了する。
- コメントが 0 件の場合: 「対応するコメントはありませんでした」と伝えて終了する。
- `source` が `user` のものが人手のレビュー指摘。`ai` / `agent` は原則対象外だが、対応が必要と
  読み取れるものは扱ってよい。迷う場合はユーザーに確認する。

### 2. プロジェクトのルールを読む

修正がプロジェクトの流儀に沿うよう、`docs/rules/` 配下の `*.md` があれば**すべて**読む。
無ければ `CLAUDE.md` や既存コードの規約（命名・構成・テストの書き方）に合わせる。
プロジェクトが規約の所有者であり、我流を持ち込まない。

### 3. コメントごとに対応する

取得した各コメントについて、順に対応する。

- 対象箇所を必ず**実際にコードを読んで**確認する（`filePath` を開き、`newRange` の行前後を見る）。
  コメントは特定の hunk への指摘なので、文面だけで判断せず該当コードと突き合わせる。
- 本文の意図を汲んで**コード修正まで実施する**。呼び出し側・テストなど**影響範囲も追って**
  まとめて直す。
- 指摘が質問・確認なら、コードを調べて回答を用意し、修正が要れば併せて行う（回答は
  ステップ 5 の一覧に含める）。
- 修正後、可能ならプロジェクトの検証を通す（lint / 型チェック / 関連テスト）。検証コマンドが
  分からなければ既存の設定（`pyproject.toml` / `package.json` 等）から読み取る。Docker が要る
  テストで Docker が動いていない等、実行できないものはその旨を記録する。
- 各コメントについて「対応した／保留した（理由）」と、実施した修正内容を記録しておく
  （ステップ 5 で一覧にする）。

**手を止めて確認する場合**: 広範囲の破壊的変更、データ移行やスキーマ変更、外部への影響、
コメントの意図が明らかに曖昧で複数解釈できる、といったケースは自動で進めず
`AskUserQuestion` で確認してから続行する。

### 4. 対応済みコメントを削除する

対応（修正または回答）が完了したコメントを hunk から削除する。

```
hunk session comment rm --repo <repo> <noteId>
```

対応しきれず保留にしたコメントは**削除しない**（残して次回拾えるようにする）。

### 5. 対応内容を一覧表示する

対応した全コメントについて、**一覧（表）**でまとめて示す。各行に少なくとも次を含める。

- 対象ファイル（と対象行）
- コメント本文（要約可）
- 実施した対応内容（修正の要点、または回答）
- 状態（対応済み / 保留＋理由）

例:

| 対象 | コメント | 対応内容 | 状態 |
|---|---|---|---|
| `path/to/file.py:112` | 戻り値を tuple ではなくデータクラスに | 戻り値をデータクラス化し呼び出し側も更新 | 対応済み |

### 6. コミットして push する

対応した変更を git にコミットして push する。

- まずコミット規約を確認する。`docs/COMMIT.md` があれば最優先で従う。無ければ
  `docs/ISSUE.md` / `docs/rules/` / `CLAUDE.md` にコミット規約がないか確認し、あれば従う。
  いずれも無ければ、何を・なぜ変えたかが分かる簡潔なメッセージにする。
- コミット前に `git status` / `git diff` で変更内容を確認する。hunk 対応と無関係な変更が
  混ざっていないかを見て、混ざっていればユーザーに確認する。
- **現在のブランチがデフォルトブランチ（main 等）の場合**は、そのままコミットせず、
  ブランチを切るか・このままコミットするかを `AskUserQuestion` で確認する。フィーチャー
  ブランチ上なら現在のブランチにコミットする。
- コミット後 `git push`（上流が未設定なら `git push -u origin <branch>`）する。push は外部への
  反映のため、失敗時（リモート拒否・コンフリクト等）は状況を伝えて指示を仰ぐ。

### 7. 完了報告

コミット（ハッシュ・メッセージ）と push 先ブランチ、ステップ 5 の対応一覧を簡潔に示す。
保留にしたコメントがあれば理由とともに再掲する。

## 補足

- コメントが多い場合も、ステップ 3〜4 を全コメントについて回してから、ステップ 5〜6 を
  一度だけ行う（コミットは論理単位で分けてよい）。
- 修正が全く不要（全て質問で回答のみ、等）の場合はコミットを作らず、その旨を報告する。

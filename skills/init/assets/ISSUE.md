# Issue 作成ルール・コミット規約

`za:issue` が issue を作るとき、`za:fix-issue` / `za:review` がコミットするときに従うルール。

## 言語

issue・コミットメッセージともに日本語で書く。コード識別子・コマンドは原語のまま。

## Issue 作成ルール

### タイトル

`{種別}: {内容}` の形式。種別は下記ラベルと対応させる（例: `feat: 設定画面を追加`）。
端的に、何をするかが分かるように。

### 本文テンプレート

```markdown
## 概要
{この issue で何を実現するか}

## やること
- {達成すべき振る舞いを箇条書き（レイヤー別手順ではなく成果で書く）}

## 受け入れ条件
- [ ] {完了と判断できる条件}

## 依存
- {先に完了が必要な issue。無ければ「なし（すぐ着手可）」}

## 参照
- 元プラン: {docs/plans/ のファイルパス}
```

### ラベル

- `feat` — 新機能
- `fix` — バグ修正
- `refactor` — 挙動を変えない改善
- `docs` — ドキュメント
- `chore` — ビルド・設定・雑務

### 分割の方針

1 つの issue が大きすぎる、または独立した複数の成果物を含む場合は、それ単体で着手・検証
できる単位（縦スライス）に分割し、順序依存は「依存」欄に明記する。

## コミット規約

`za:fix-issue` / `za:review` はこの規約でコミットする。

### メッセージ形式

```
{種別}: {要約}

{本文：何を・なぜ変えたか。必要なら}

{対応 issue があれば: 例) #42}
```

- **種別**: `feat` / `fix` / `refactor` / `docs` / `test` / `chore`（issue ラベルと対応）。
- **要約**: 日本語で簡潔に。命令形で「〜する」調に統一。
- **issue 参照**: 対応する issue があれば本文に `#番号` を含める。「Closes #番号」は PR 側で使う。
- **粒度**: 1 コミット 1 論理変更。複数の関心事が混ざるなら分割する。

### 例

```
feat: 設定画面を追加する

メニューから開ける設定画面を追加。表示・保存の基本動作を実装する。

#12
```

# za:merge — PR をマージしてデフォルトブランチに戻す

レビューが完了した GitHub プルリクエスト（PR）をマージし、ローカルをデフォルトブランチに
戻して最新化する。狙いは、開発サイクルの締めとして PR を安全に取り込み、次の作業に
すぐ移れる状態にすること。積み残しがあれば issue として残し、忘れずに拾えるようにする。

マージは外部に対する取り消しの効きにくい操作なので、**マージ前に状態を確認し、ユーザーの
承認を得てから実行する**（後述）。下の手順を順番に実行する。

## 手順

### 1. 対象 PR を特定する

マージ対象は、**このスキルに渡された引数の PR 番号/URL** から特定する（例: `/za:merge 42`）。

- 引数が無い場合: まず現在ブランチに紐づく PR を `gh pr view --json number,title,url` で
  探す。見つからなければ `gh pr list` を提示し、「どの PR をマージしますか？」と確認する。
- `gh pr view <番号>` で PR の状態（タイトル・base←head・レビュー・チェック）を把握する。
- PR が見つからない／既にクローズ・マージ済み、`gh` が使えない場合は、推測で進めず状況を
  伝えて確認する。

### 2. マージ可能な状態かを確認する

レビュー完了後の取り込みなので、マージしてよい状態かを確認する。
`gh pr view <番号> --json state,mergeable,mergeStateStatus,reviewDecision,statusCheckRollup`
などで次を見る:

- **レビュー承認**: `reviewDecision` が承認相当か。未承認・変更要求が残っていれば、その旨を
  伝えて、マージしてよいか確認する（勝手に押し切らない）。
- **CI / ステータスチェック**: 失敗・保留のチェックがないか。失敗があれば内容を伝え、
  マージするか確認する。
- **コンフリクト**: `mergeable` が衝突なしか。衝突していればマージできないので、解消が
  必要な旨を伝えて止まる（このスキルでは勝手に解消しない）。

問題があるのに進める場合は、必ずユーザーの明示的な了承を得る。

### 3. マージ方法を決める

PR のマージ方法（merge commit / squash / rebase）と、head ブランチを削除するかを決める。

- `docs/PR.md` にマージ方法の指定（squash 必須など）があればそれに従う。
- 指定が無ければ、リポジトリの慣習（既存 PR のマージ履歴など）に合わせる。判断がつかない
  場合は **squash を既定の推奨**として、どの方法でマージするかユーザーに確認する。
- 取り込み後の head ブランチは原則削除する（`--delete-branch`）。残す指定があれば従う。

### 4. ユーザーに提示して承認を得る

マージ予定の内容を提示する。最低限:

- 対象 PR（番号・タイトル）と base ← head
- レビュー・CI・コンフリクトの状態（手順2の結果）
- マージ方法（squash / merge / rebase）と head ブランチ削除の有無

「この内容でマージしてよいか」を確認する。
（ユーザーが事前に「確認不要、そのままマージ」と明示している場合は省いてよい。）

### 5. マージを実行する

承認後、`gh pr merge` でマージする。

```sh
gh pr merge <番号> --squash --delete-branch   # 方法は手順3で決めたものに置き換える
# merge commit なら --merge、rebase なら --rebase
```

- マージに失敗した場合（保護ルール・チェック未達・権限など）は、強制的な回避策を取らず、
  失敗内容をそのまま報告して確認する。

### 6. デフォルトブランチに戻して最新化する

マージ後、ローカルをデフォルトブランチに戻す。

- デフォルトブランチを確認する
  （`gh repo view --json defaultBranchRef -q .defaultBranchRef.name`）。
- そのブランチへ切り替えて最新を取り込む: `git switch <default>` → `git pull`。
- 作業ツリーに未コミットの変更があると切り替えに失敗する。その場合は無理に進めず、状況を
  伝えて確認する。
- `--delete-branch` でリモートの head は消えるが、ローカルの作業ブランチが残っていれば、
  不要なら削除を提案する（`git branch -d <head>`）。

### 7. 積み残しタスクを issue 化して案内する

今回の PR で対応しきれなかったこと（レビューで「対応しない」と判断した指摘、TODO/FIXME、
スコープ外として送った改善、フォローアップ）があれば、**issue として残すことを案内する**。

- 積み残しの候補を一覧で示す（何を・なぜ今回やらなかったか）。
- 残すものについては issue 化を提案する。プランファイル起点であれば `/za:issue` を、
  単発であれば `gh issue create`（`docs/ISSUE.md` のルールに従う）を案内する。
- ユーザーが「作って」と言えば作成まで行い、判断を委ねる場合は候補の提示にとどめる
  （勝手に大量の issue を作らない）。
- 積み残しが無ければ「積み残しなし」と報告する。

### 8. 完了報告

次を報告する:

- **マージ結果**: マージした PR（番号・タイトル）・マージ方法・削除した head ブランチ
- **現在の状態**: 切り替えたデフォルトブランチと最新化（pull）の結果
- **積み残し**: issue 化した／案内した内容、または「積み残しなし」

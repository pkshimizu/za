# za:ship — issue を実装して PR・レビューまで通しで実行

GitHub issue を起点に、次の 3 スキルをこの順で**途中の確認なしに**通しで実行する。
狙いは、issue 番号ひとつを渡せば「レビュー収束済みの PR」まで無人で到達させ、
ユーザーの待ち時間・往復回数を最小にすること。

1. `za:fix-issue` — issue の実装とコミット
2. `za:pr` — PR の作成
3. `za:review` — 観点別の並列レビューと修正・push

マージは**行わない**（マージは取り消しの効きにくい操作であり、最終確認はユーザーに
残すため）。完了時に `/za:merge` を案内する。

## 実行方法

各ステップは対応するスキルを **Skill ツールで順に呼び出して**実行する
（`za:fix-issue` → `za:pr` → `za:review`）。各スキルの手順・参照ドキュメント
（`docs/rules/`・`docs/PR.md`・`docs/review-perspectives/` など）はそのスキル自身の
指示に従う。この通しフローが上書きするのは「ユーザー確認の扱い」だけ（下記）。

- `za:pr` に渡す変更内容は手順 1 の結果（ブランチ・コミット）から拾う。
- `za:review` に渡す PR 番号は手順 2 で作成した PR の番号を使う。

## ユーザー確認の扱い（このフローでの上書きルール）

各スキルの手順には「ユーザーに確認する」「承認を得る」というステップがあるが、
この通し実行では**確認のために停止しない**。代わりに次のルールで進める:

- **推奨案がある確認**（PR 作成の承認、実装方針の選択など）: 自分が推奨する案を
  そのまま採用して先へ進む。何をどう判断したかを**判断ログ**（下記）に記録する。
- **推奨案を立てられる曖昧さ**（issue の記述が曖昧、複数の妥当な実装がある等）:
  issue 本文・既存コード・プロジェクトのルールから最も整合する解釈を選んで進め、
  判断ログに「他にあり得た解釈」も添える。ユーザーは最後のログを見て、ずれていれば
  修正を指示できる。
- **レビューの指摘対応**: `za:review` の「対応する/しない」の判断も自動で行い、
  対応しなかった指摘は理由つきでログに残す（これは za:review 自体のサマリーと同じ）。

ただし、次の状況は**判断ではなく前提の不成立**なので、自動で進めず停止して報告する:

- 指定された issue が見つからない・クローズ済み・引数が無い
- 作業ツリーに関係のない未コミット変更がある（混入を防ぐため）
- 現在のブランチがデフォルトブランチでなく、分岐元の判断がつかない
- push・PR 作成・CI などが失敗し、自動リトライで解消しない

これらは推測で進めると取り返しのつかない混入や誤実装につながるため、確認省略の
対象にしない。

## 判断ログ

通し実行中に自動採用した判断を、その場で記録して最後にまとめて出力する。形式:

```
## 自動採用した判断
| # | 場面 | 判断 | 理由 / 他にあり得た選択肢 |
|---|------|------|--------------------------|
| 1 | 実装方針 | 稼働時間は全置換APIにした | 既存の photo/work_location と同型のため。差分更新も可能だったが… |
| 2 | PR 作成 | 承認確認を省略して作成 | 通し実行の方針どおり |
```

判断が一つも無ければ「自動採用した判断: なし」と報告する。ログは会話への出力のみで、
ファイルには残さない（PR・コミットに履歴は残るため）。

## 完了報告

最後に次をまとめて報告する:

- **成果物**: ブランチ名・コミット一覧・PR の URL とタイトル
- **レビュー結果**: 対応した指摘 / 対応しなかった指摘（理由つき）のサマリー
  （za:review のサマリーを流用してよい）
- **自動採用した判断**: 上記の判断ログ
- **次の作業**: マージは `/za:merge <PR番号>` で行えることを案内する

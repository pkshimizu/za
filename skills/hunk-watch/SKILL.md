# za:hunk-watch — hunk のレビューコメントを監視して自動対応

[hunk](https://www.hunk.dev/) は「レビュー起点の diff ビューア」で、diff の特定の hunk / 行に
インラインでレビューコメントを付けられる。このスキルは、そのコメントを**継続監視**し、
新しいコメントが付いたら**内容を解釈してコード修正まで自動で行い**、対応が済んだコメントを
**自動削除**する。これをユーザーが止めるまで繰り返す常駐フロー。

狙いは、レビューで気づいたことをコメントするだけで、その場で実装まで反映されること。

## 前提と制約（先に理解する）

- **コメントは起動中の hunk セッション内にのみ存在する。** `hunk diff` / `hunk show` などで
  開いた TUI（またはそのデーモン）が生きている間だけ `hunk session comment list` で取れる。
  **セッションを閉じるとコメントは破棄される**ため、監視対象のセッションは開いたままにしてもらう。
- 監視は**ポーリング**（プッシュ通知ではない）。「即座」はポーリング間隔（既定 5 秒）で決まる。
- 対応方針は固定で **「自動実装 → 対応済みコメントを自動削除」**。破壊的な変更や判断が大きく
  割れる指摘に当たった場合だけ、いったん手を止めてユーザーに確認する（下記 4 の但し書き）。
- このスキルは**コミットしない**（作業ツリーへの修正まで）。コミット/PR は別スキル（/za:pr 等）。

## 手順

### 0. 監視対象リポジトリを確認する

対象は原則カレントリポジトリ（`git rev-parse --show-toplevel`）。ユーザーが別パスを指定した
場合はそれを使う。監視を始める前に、対象リポジトリで hunk セッションが動いているかを
`hunk session list` で確認する。

- セッションが無い場合: 「対象リポジトリで hunk を起動（例: `hunk diff` や `hunk show`）して
  コメントできる状態にしてください。起動後、コメントを待ち受けます」と伝える。セッションが
  無くても監視ループは開始してよい（コメントが取れないうちは待機し続け、セッションが立ち上がって
  コメントが付いた時点で拾う）。

### 1. 新規コメントを待ち受ける（バックグラウンド監視）

このスキルディレクトリに同梱の監視スクリプトを **`run_in_background: true`** で起動する。

```
bash <このスキルの assets>/wait-for-comment.sh <repo> [interval_sec=5] [timeout_sec=1800]
```

このスクリプトは、未解決の user コメントが現れるまでブロックし、現れたらコメント配列を
JSON（`noteId` / `filePath` / `newRange` / `body` / `hunkIndex` を含む）で出力して終了する。
プロセスが終了すると、その出力を受け取って対応を始める（コメントが無いままタイムアウトした
場合は `[]` を出して終了するので、その時は再度ステップ 1 に戻って待ち受け直す）。

バックグラウンドにするのは、待ち受け中に他の作業やユーザーとの対話をブロックしないため。

### 2. 受け取ったコメントを対応対象として整理する

出力 JSON の各コメントについて、対象ファイル（`filePath`）・対象行（`newRange`）・
本文（`body`）・識別子（`noteId`）を把握する。複数同時に付いていれば、この回でまとめて扱う。

対象箇所を必ず**実際にコードを読んで**確認する（`filePath` を開き、`newRange` の行前後を見る）。
コメントは特定の hunk への指摘なので、文面だけで判断せず該当コードと突き合わせる。

### 3. プロジェクトのルールを読む

修正がプロジェクトの流儀に沿うよう、`docs/rules/` 配下の `*.md` があれば読む。無ければ
`CLAUDE.md` や既存コードの規約（命名・構成・テストの書き方）に合わせる。プロジェクトが
規約の所有者であり、我流を持ち込まない。

### 4. コメントごとに対応する

各コメントについて、本文の意図を汲んで**コード修正まで実施する**。

- 指摘が具体的な修正要求（例: 「tuple ではなくデータクラスに」）なら、その通り実装する。
  呼び出し側・テストなど**影響範囲も追って**まとめて直す。
- 指摘が質問・確認（例: 「ここは意図通り？」）なら、コードを調べて回答を用意し、修正が要れば
  併せて行う。回答は最後の報告（ステップ 6）に含める。
- 修正後、可能ならプロジェクトの検証を通す（lint / 型チェック / 関連テスト）。この環境では
  DB 依存テストは Docker 経由（`docker compose run --rm api ...`）で、Docker が動いていなければ
  その旨を報告し、少なくとも lint と構文/インポートの健全性は確認する。

**但し書き（手を止めて確認する場合）**: 次に当たったら自動で進めず `AskUserQuestion` で確認する。
広範囲の破壊的変更、データ移行やスキーマ変更、外部への影響、コメントの意図が明らかに曖昧で
複数解釈できる、といったケース。確認が済んだら続行する。

### 5. 対応済みコメントを削除する

そのコメントへの対応（修正または回答）が完了したら、hunk から削除して解決済みにする。

```
hunk session comment rm --repo <repo> <noteId>
```

削除により同じコメントを次のポーリングで再検知しなくなる（無限ループ防止も兼ねる）。
対応しきれず保留にしたコメントは**削除しない**（次回拾い直せるように残す）。

### 6. この回の対応を報告する

この回で対応したコメントについて、コメント本文・対象ファイル・実施した修正（または回答）・
検証結果を簡潔に報告する。保留にしたものがあれば理由も添える。

### 7. 監視を続ける

ステップ 1 に戻り、次の新規コメントを待ち受ける。これをユーザーが止める（中断）まで繰り返す。
ユーザーから「監視を止めて」等の指示があれば、待ち受けを終了し、これまでの対応をまとめて報告する。

## 補足

- 一度に大量のコメントが付いた場合も、ステップ 2〜6 を各コメントについて回して順に片付ける。
- hunk セッションが途中で閉じられた（`hunk session list` が空になった）場合は、待ち受けは
  継続しつつ「セッションが閉じられたようです。コメントするには hunk を開き直してください」と
  一度知らせる。

# za:resolve-issue — issue の不明点を整理し対応方針を確定する

GitHub issue を実装可能な状態に「詰める」。狙いは、着手前に issue の曖昧さ・未決定事項を
洗い出してユーザーと方針を確定し、その結果を issue に残すこと。ただし今は決められない点は
無理に決めず「保留」として明示し、先送りできるようにする。**このスキルでは実装しない**
（実装は za:fix-issue）。

下の手順を順番に実行する。各ステップの「なぜ」を理解した上で、状況に合わせて判断すること。

## 手順

### 1. 対象の issue を取得する

対象は、**このスキルに渡された引数の issue 番号**から特定する（例: `/za:resolve-issue 42`）。
`gh issue view <番号> --comments` で、タイトル・本文・コメント・ラベルを取得し、何を
やろうとしている issue かを把握する。

- 引数が無い場合: `gh issue list`（オープンな issue）を提示し、「どの issue を詰めますか？」と
  ユーザーに確認する。
- 番号の issue が見つからない／クローズ済みの場合: 推測で別の issue を扱わず、状況を伝えて確認する。
- リポジトリが git でない、`gh` で issue を取得できない場合も、その旨を伝えて確認する。
- コメントに過去の resolve-issue 記録（後述の「対応方針」）があれば、確定済み・保留中の
  状態として踏まえ、蒸し返さない。

### 2. プロジェクトの文脈を読む

`docs/CONTEXT.md` を読む。プロジェクトのドメイン・用語・方針に沿って不明点を判断するため。

- 無い場合: 「`docs/CONTEXT.md` が無いため、コードから文脈を補って進めます」と一言伝え、
  リポジトリ構成をざっと見て補う。処理自体は止めない。

### 3. 不明点・未決定事項を洗い出す

issue の本文・コメントとコンテキストを突き合わせ、**実装に入る前に決めておくべき点**を
列挙する。典型的な観点:

- スコープ（何を含み、何を含まないか）・受け入れ条件の曖昧さや抜け
- 挙動・UX の選択（キー割り当て、表示、デフォルト、エッジケースの扱い）
- 影響する／再利用する既存モジュール・データ・API、依存する他 issue
- データ構造・インターフェース・技術選定の選択肢
- 非機能要件（性能・セキュリティ・後方互換性）

### 4. まず調査し、残りをユーザーに確認する

不明点ごとに、**まずコードや既存ドキュメントを調べて自力で解消できないか試す**
（ユーザーの手間を最小化するため。調べれば分かることは質問しない）。

- 既存実装・命名規約・テスト・設定を調べて答えが出るものは確定させ、「調査で解消した点」
  として記録する。
- 判断が**ユーザーに委ねられる**もの（仕様の意図、優先順位、好みの選択）だけを、
  **番号付きの一覧でまとめて**質問する。各項目に**自分の推奨案**を添える。
- 各項目には、**「保留（今は決めない）」の選択肢を必ず残す**。情報が足りない・後で決めたい点を
  ユーザーが先送りできるようにする。保留を選んだ項目には、**何が決まれば判断できるか
  （保留の理由・再検討のトリガー）**を一言添える。
- 「全部おまかせ（推奨案で確定）」「全部保留（今は詰めない）」もまとめて選べるようにする。

回答を得たら、新たな不明点が出ていないか確認し、必要なら繰り返す。

### 5. 方針を整理する

回答を踏まえ、論点を **「確定した方針」** と **「保留（未決定）」** に仕分ける。

- 確定分は、issue の受け入れ条件・詳細に落とせる粒度（何をどうするか）で言語化する。
- 保留分は、再検討の条件（何が決まれば判断できるか）を添えて残す。無理に決めない。

### 6. issue に記録する

確定した方針と保留事項を、対象 issue に**コメントとして追記**する。本文を書き換えず履歴を
残せること、`za:fix-issue` が `--comments` で参照できることから、コメントを使う。

投稿は外部への書き込みなので、**投稿内容をユーザーに提示して承認を得てから**
`gh issue comment <番号> --body-file -`（ヒアドキュメント）で投稿する。書式は末尾の
「デフォルトの記録書式」に従う。

- 何も確定せず**すべて保留**になった場合は、無理にコメントせず、その旨と再開条件を報告して
  終えてもよい（または最小限の「保留メモ」コメントにとどめる）。
- 確定により issue のスコープ・受け入れ条件が大きく変わる場合は、本文の更新を提案する
  （勝手に本文を上書きせず、ユーザーに確認してから）。

### 7. 完了報告

確認した論点とその結論、保留事項（と再検討の条件）、投稿したコメントの URL を簡潔に伝える。
実装に進めるなら、次は `/za:fix-issue <番号>` を案内する。まだ保留が多く実装に入れない場合は、
何が決まれば着手できるかを明示する。

---

## デフォルトの記録書式

issue へ追記するコメントの標準テンプレート。

```markdown
## 対応方針（resolve-issue）

### 確定した方針
- {論点}: {決定内容}（理由: {根拠}）

### 保留（未決定）
- {論点}: 保留。{再検討の条件・何が決まれば判断できるか}

### スコープ／受け入れ条件への反映
- {確定により本文に反映すべき点があれば箇条書き。無ければ「なし」}
```

# za:review — PR を観点別に並列レビューして修正

GitHub プルリクエスト（PR）を、プロジェクトが定めたレビュー観点ごとに**並列で**レビューし、
出た指摘を修正し、指摘が出なくなるまでレビューを繰り返す。狙いは、観点を分担させて
網羅的かつ素早くレビューし、その場で直し切ること。そしてレビューで得た知見をルールとして
蓄積し、同じ指摘が二度出ないようにすること。

下の手順を順番に実行する。修正はコミットするが、**push は行わない**（必要なら最後に案内する）。

## 手順

### 1. 対象 PR を特定してチェックアウトする

レビュー対象は、**このスキルに渡された引数の PR 番号/URL** から特定する
（例: `/za:review 42`）。

- 引数が無い場合: `gh pr list` を提示し、「どの PR をレビューしますか？」と確認する。
- `gh pr view <番号>`（必要なら `--comments`）と `gh pr diff <番号>` で内容と差分を把握する。
- 修正を加えるため、`gh pr checkout <番号>` で PR のブランチをローカルに取り出す。
  作業ツリーに未コミットの変更があれば、混ざらないよう先に確認する。
- PR が見つからない／クローズ済み、`gh` が使えない場合は、推測で進めず状況を伝えて確認する。

### 2. レビュー観点を読み込む

`docs/review-perspectives/` 配下の `*.md` を**すべて**読み込む。各ファイルが 1 つの
レビュー観点（例: `security.md`, `readability.md`, `tests.md`）で、何をどう見るかが
書かれている。レビューはこの観点に沿って行う（プロジェクトが観点の所有者）。

- ディレクトリやファイルが無い場合: 「`docs/review-perspectives/` が無いため、標準観点で
  レビューします」と伝え、末尾の**「デフォルトのレビュー観点」**を使う。

### 3. 観点ごとに並列でレビューする

観点ごとに**サブエージェントを 1 つ起動し、すべて並列で実行する**（同時に投げて待つ）。
観点を分担させることで、各エージェントが 1 つの視点に集中でき、レビューが深く・速くなる。

各エージェントには次を渡す:

- 担当する観点ファイルの内容（その観点の見方）
- PR の差分（`gh pr diff` の結果）と、必要に応じてリポジトリを読む指示
  （差分だけでは文脈が足りないことがあるため、周辺コードの参照を許可する）
- 出力形式（下記「指摘の形式」）

各エージェントは、その観点での指摘の一覧を構造化して返す。指摘が無ければ「指摘なし」を返す。

### 4. 指摘を集約して修正する

全エージェントの指摘を集約し、重複をまとめ、重要度順に整理する。各指摘について
**対応するか否かを判断**する:

- 対応するもの: コードを修正する。観点間で矛盾する指摘があれば、プロジェクトの方針
  （観点ファイル・既存コード・ADR 等）に照らして優先順位を決め、判断の根拠を残す。
- 対応しないもの: 理由（誤検知、スコープ外、設計上の意図、トレードオフ 等）を記録する。
  これは後のサマリーと、再レビューでの蒸し返し防止に使う。

### 5. 再レビューして収束させる

修正後、変更点を中心に**もう一度（観点ごとに並列で）レビュー**し、新たな指摘が無くなるまで
4→5 を繰り返す。

- すでに「対応しない」と理由つきで判断した指摘が再度出ても、蒸し返さない
  （堂々巡りを避けるため）。
- 新たに対応すべき指摘が無くなったら収束とみなす。瑣末な指摘で延々と往復しそうな場合は、
  3 周程度を目安に切り上げ、残りは「対応しなかった指摘」として扱う。

### 6. コミットする

修正が完了したら、変更内容を確認（`git status` / `git diff`）してコミットする。
コミット規約が `docs/ISSUE.md` や `docs/rules/` にあればそれに従う。無ければ、レビュー対応で
何を直したかが分かる簡潔なメッセージにする。

### 7. 修正内容のサマリーを表示する

次を表示する:

- **修正内容のサマリー**: 何を直したか（ファイル単位の要点）
- **対応した指摘**: 観点・指摘・どう直したか
- **対応しなかった指摘**: 観点・指摘・**対応しなかった理由**

対応/未対応は一覧（表など）で見やすく示す。

### 8. 繰り返しそうな指摘をルール化する

今回の指摘のうち、**今後も繰り返し指摘される可能性が高いもの**（プロジェクト共通の規約・
落とし穴）を `docs/rules/{name}.md` に追記・修正する。`{name}` は内容に合った分類
（例: `naming.md`, `error-handling.md`）。既存の近いルールがあればそこへ追記し、無ければ
新規作成する。これにより `za:fix-issue` など他の作業が同じルールを参照でき、同種の指摘が
減っていく。

- ルールを更新した場合は、その内容も手順6のコミットに含め、サマリーで何を追加・修正したか
  報告する。

## 指摘の形式

各レビューエージェントは、指摘を次の形式で返す:

```
- 重要度: blocker | major | minor | nit
  場所: <ファイルパス>:<行>
  指摘: <何が問題か>
  根拠: <なぜ問題か。観点ファイルや規約・一般原則への参照>
  提案: <どう直すか>
```

## デフォルトのレビュー観点

`docs/review-perspectives/` が無い場合に使う標準観点。これらを並列エージェントの担当とする:

- **正確性/バグ**: ロジック誤り、境界条件、null/例外、競合、後方互換性
- **可読性/保守性**: 命名、責務分離、重複、過度な複雑さ、既存コードとの一貫性
- **テスト**: 変更に対するテストの有無・妥当性、抜けているケース
- **セキュリティ**: 入力検証、認可、機微情報の扱い、インジェクション
- **パフォーマンス**: 不要な計算・I/O、N+1、明らかな非効率
